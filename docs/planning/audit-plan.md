# 交易分析系统持续审核蓝图

> **文档类型**：长期审核蓝图（持续维护，非一次性审核结论）  
> **适用范围**：行情与分析流水线、LLM 决策链、交易授权、报告、回放/回测、UI 与运行安全。  
> **边界**：本蓝图定义审核方法、证据门槛和验收条件；不替代架构事实来源，不记录某次审核结论，也不授权在审核中修改代码或执行交易。

## 1. 目的、范围与事实来源

持续审核的目标是提升交易分析系统的可靠性、可解释性和决策准确性：任何展示给用户的方向、价位、风险与叙事，都应能追溯至相同时间点的数据、明确的计算事实和 Manager 的授权决定。

本蓝图与下列文档配合使用，而不重复维护其具体实现事实：

- [架构健康检查](../architecture/review.md)：模块边界、可读性与长期架构风险。
- [报告可信度架构](../architecture/report-trust.md)：事实注册、证据溯源、不变量和可靠度语义。
- [LLM Agents 架构](../architecture/llm-agents.md)：阶段职责、输入输出与编排。
- [测试策略](../testing/strategy.md)：测试分层、命令和发布门禁。
- [回测架构](../architecture/backtesting.md)：point-in-time、回放与回测边界。
- [开发者速查表](../reference/cheat-sheet.md)：模块、入口和常用测试定位。

审核默认只读：不得格式化、修改、暂存、提交或推送项目文件；不得触发 MT5 下单；不得记录、打印或上传凭据。审核发现与修复工作分离，只有证据完整的根因才可进入 Issue。

## 2. 通用审核方法与证据门槛

每次审核均按 **Ground → Evidence → Decision** 顺序完成，不能以测试绿灯、模型文本看似合理或最近一次运行成功代替证据。

| 阶段 | 必须完成的工作 | 可交付物 |
| --- | --- | --- |
| Ground | 冻结启动快照：分支、HEAD、`git status`、审核时间、目标归档/基线与数据 as-of；探索入口、配置、调用链和现有测试。 | 范围声明、基线可用性结论、受影响闭包。 |
| Evidence | 为每项结论收集文件/符号/行号、运行产物、结构化输出、测试结果或最小复现；标明数据、传输、解析、验证、兼容或代码层。 | 可复核证据链；推测只能标为“设计风险”或“未验证边界”。 |
| Decision | 逐项确认触发条件、影响、严重度、最小整改方向和验收测试；按根因去重。 | Runtime Ledger 摘要，以及证据充分时的 Issue。 |

### 2.1 失败分级

| 级别 | 判定标准 | 审核处置 |
| --- | --- | --- |
| P0 | 可导致未经授权交易、未来数据泄漏、凭据泄露、或高概率造成严重错误执行。 | 阻断发布/执行；立即记录可复现证据。 |
| P1 | Manager 授权、交易几何、方向、`signal_id`、报告叙事或关键风险控制不一致，可能误导交易决策。 | 阻断受影响功能发布；创建或更新根因 Issue。 |
| P2 | 明确降低可靠性、可观测性、回放兼容性或降级可信度，但存在保护或影响受限。 | 纳入修复队列；创建或更新根因 Issue。 |
| P3 | 改进项、测试覆盖缺口或尚无最小复现的设计风险。 | 写入 Runtime Ledger；不以推测创建缺陷 Issue。 |

环境阻断、外部 API 拒绝、网络代理差异、缺少归档或证据不全必须单独标注，不能被表述为代码缺陷。反之，失败不能被静默回退伪装为成功。

## 3. 审核运行节奏与升级规则

每周与发布前审核是流程门槛，不在本蓝图中创建新的定时任务。三类审核的输入、输出和可比性不同：

| 节奏 | 输入与允许操作 | 输出 | 不可比/升级或阻断规则 |
| --- | --- | --- | --- |
| 每日：最新归档深度审核 | **仅**加载最后一次可加载的回放归档及其阶段产物；零实时 API 调用、零新增 LLM 调用与费用。 | 归档完整性、决策链一致性、离线回放/门禁结果、Runtime Ledger。 | 归档损坏、schema 不兼容、关键阶段缺失或启动快照不可定位时，记录为审核边界；不得自行以实时请求替代归档。 |
| 每周：受控真实 LLM 抽样 | 固定或受控样本、明确模型/配置和费用上限；只做必要的真实 LLM 漂移、超时、结构化输出与回退抽样。 | 阶段级延迟/来源/解析/置信度比较与漂移结论。 | 输入、模型、提示词、schema、数据 as-of 或采样策略变化时不可直接将差异判为模型回归；先标明比较条件。 |
| 发布前：完整门禁 | 完整离线门禁、受影响场景测试、归档/回放兼容验证，并进行受控真实 LLM 抽样。 | 发布审核记录、阻断项和剩余边界。 | P0/P1、关键归档不可重放、授权链未验证或真实 LLM 抽样无有效证据时阻断相关发布。 |

每日审核以“最后一次可加载归档”为对象，不自动把“最近一次运行”或“最近一次成功运行”视为可用基线。若需要比较漂移，应另外确认一个状态成功、证据完整、版本可读且 HEAD 可定位的成功基线，并明确哪些字段可比。

## 4. 模块化审核计划

每个模块均必须形成六类记录：**审计目标、关键不变量、证据来源、离线测试/最小复现、失败分级、未验证边界**。以下矩阵定义最低检查面；项目实现和测试入口以引用的架构及测试文档为准。

### 4.1 数据与时序

- **审计目标**：保证行情、多周期数据、DXY/US10Y、新闻、日历和社交数据均在可声明的 as-of 时点可用，且市场开闭状态、缺失与回退语义一致。
- **关键不变量**：任何分析/报告事实不得晚于其决策 as-of；旧新闻或日历事件不得被描述为未来风险；来源失败、空数据和缓存/回退必须可见；同一字段的来源和时间语义不能冲突。
- **证据来源**：归档元数据、数据源状态、时间戳、市场状态、fetch/解析错误、报告的 freshness/reliability 字段及相应日志脱敏摘要。
- **离线测试与最小复现**：构造过期报价、未来事件、缺失 DXY/US10Y、空新闻、来源时间不一致和回退输入；验证数据契约、时效门禁和降级展示。
- **失败分级**：未来数据泄漏、把不可交易市场当作可执行为 P0/P1；未透明的陈旧/缺失数据为 P1/P2；尚未覆盖的来源边界为 P3。
- **未验证边界**：每日归档审核不能证明外部站点当前可访问或实时数据正确，只能验证归档中已记录的事实与失败语义。

### 4.2 指标与市场结构

- **审计目标**：验证多周期 OHLC、ICT/PA、Session PA、FVG/OB、趋势、流动性与结构事实的计算契约、索引和点位可追溯性。
- **关键不变量**：结构结论必须关联可定位的时间框架、K 线/索引与价格；不同周期不得错位；FVG/OB 与趋势/流动性事实不能越过 as-of；报告引用的点位必须能回溯至已验证事实。
- **证据来源**：输入 OHLC 摘要、结构分析产物、事实注册表、图表/报告字段、计算函数与针对性断言。
- **离线测试与最小复现**：使用包含边界 K 线、缺口、会话切换、相同高低点、无效索引和多周期对齐偏移的固定夹具；回归已知结构/价位。
- **失败分级**：可直接生成错误方向或交易价位的结构错位为 P1；显示与计算不一致、不可追溯或保护性降级失效为 P2；覆盖缺口为 P3。
- **未验证边界**：离线计算验证不保证指标在真实市场中的预测能力；预测准确性应与计算正确性分开记录。

#### 4.2.1 技术分析专项审核

技术分析专项把“指标算出了值”与“该值可在当时被系统知晓、可被后续正确引用”分开验证。每次专项审核至少覆盖下表中的项目；审计记录必须保存使用的固定 OHLC 夹具、时区、时间框架和 as-of 截点。

| 检查面 | 必须核对的内容 | 离线对照与最小复现 | 判定边界 |
| --- | --- | --- | --- |
| OHLC 与时间轴 | Open/High/Low/Close/Volume 的字段类型、去重、升序索引、时区、缺口、未收盘 K 线和多周期重采样边界。 | 空 K 线、重复时间戳、乱序索引、DST/会话切换、未收盘末根和不同周期对齐夹具。 | 用到未来 K 线、把未收盘 K 线作为已确认结构为 P0/P1；可见性标记丢失为 P2。 |
| 指标计算契约 | 参数、warm-up 长度、NaN 传播、舍入/tick 精度、分母为零与极值处理；同一输入的确定性。 | 使用手工可验算的小序列和固定黄金夹具，对照预期 SMA/EMA、ATR、RSI、趋势或项目实际指标输出。 | 影响入场/止损/目标的数值错误为 P1；边界值或显示精度偏差为 P2。 |
| 结构与形态 | swing、BOS/CHoCH、流动性、FVG、OB、Session PA 的确认 K 线、索引、方向、区间和失效条件。 | 对每种形态构造最小正例、反例、相邻重叠、相同高低点和突破 K 线夹具；验证不把突破 K 线误存为结构区域。 | 错把未确认或未来结构传播为交易事实为 P1；保留/显示池不一致为 P2。 |
| 多周期与会话 | 高周期事实在低周期中的可见起点、重采样标签、交易日边界、周末/休市和 Asia/London/New York 会话。 | 在跨日、跨周、夏令时和缺失高周期 bar 的夹具上重放，比较每根低周期 K 线可见的高周期状态。 | 高周期未来值前视为 P0/P1；会话归属或标签偏移为 P2。 |
| 点位到计划传播 | 结构价位、现价、入场、SL/TP、R:R 与图表、`validated_plans`、ManagerDecision、Narrative 的来源和数值一致。 | 从单一结构事实生成候选计划，分别注入价位错位、方向反转、目标乱序和 tick 舍入，验证门禁拒绝。 | 未授权或错误几何被展示为可执行为 P1；无法追溯的点位为 P2。 |

专项审核应额外记录两种独立结论：**计算正确性**（固定输入下是否符合计算契约）和**历史/预测表现**（在严格 point-in-time 回测下的结果）。后者不能由单次归档或指标单元测试推断；样本期、交易成本、滑点、重叠信号和样本外区间不同，则不能直接比较表现。

#### 4.2.2 日内执行质量回放（情景矩阵、结构、触发与失效）

本节审计的不是“预测最终涨跌”，而是某个报告在其 **as-of** 时刻，是否有足够证据把一个候选区呈现为可执行的日内计划。它是每日最新归档深度审核的必做项；不得用“SL/TP 方向正确、R:R 合格”替代。

冲量 K 线只是需要覆盖的一个例子，不能成为唯一规则。每个获批或主展示计划都必须经过下面的**日内决策情景矩阵**；每一行都要得出 `通过`、`降级为条件计划/观察`、`阻断` 或 `证据不足`。一个维度的高分、LLM 共识或漂亮的 R:R 都不能抵消另一个必要维度的失败。

| 决策维度 | 每日审核的通用问题 | 必要证据 | 失败后的默认处置 |
| --- | --- | --- | --- |
| 快照与可见性 | 使用的是哪一根已收盘 K 线、哪个时区、哪份行情/外部数据？是否存在数据缺口、陈旧、重采样错位或未来泄漏？ | as-of、OHLC 原始索引、市场状态、数据源/回退和已收盘标志。 | 不可执行或未来数据：阻断；时区/数据无法确定：证据不足。 |
| 市场状态 | 当时是趋势、区间、压缩、扩张、反转、事件驱动还是流动性稀薄？计划的趋势/均值回归/突破类型是否适合该状态？ | 多周期趋势/ADX、波动与成交量状态、近期高低点、Session 与事件窗口。 | 策略类型与状态冲突：降级；状态事实被错误陈述：P1。 |
| 位置与结构 | 现价/入场区相对价值区、关键流动性、FVG/OB、swing、前高低和会话边界在哪里？同一区间的多空结构是否冲突？ | 有来源和方向的结构区间、距离、触及次数、确认/失效规则。 | 结构冲突未裁决：条件计划；伪造或错配结构：P1。 |
| 路径与反证 | 最近的冲量、假突破、扫流动性、回测、失败回收、BOS/CHoCH 对计划是支持还是反证？替代路径是什么，什么事实会推翻主假设？ | 已收盘 K 线序列、body/range 相对 ATR、成交量、突破后路径和明确反证条件。 | 声称的路径没有 K 线证据：P1；反证存在但未披露：P1/P2。 |
| 触发与时机 | 计划要求的回踩、拒绝、reclaim、收盘确认、结构转折、成交量或时间窗口是否已发生？当前是候选、观察、可触发还是失效？ | `status`、`trigger_confirmed`、触发证据 ID、最后已收盘 5m/15m K 线及当前价相对入场区。 | 未触发：只能是条件计划；被描述为立即可执行：P1。 |
| 失效与风险几何 | SL 是否使执行周期假设失效，是否越过相关结构/价值区并具有波动、点差、滑点可见性？TP、R:R、仓位和持仓时间是否匹配该假设？ | SL/TP、执行周期 swing/VA、5m/15m ATR、tick/spread（可得时）、风险 scale 与预期持有窗口。 | 结构内 SL、负/伪 R:R 或风险理由不成立：P1；成本或波动证据缺失：P2/观察。 |
| 流动性与执行摩擦 | 该时段的市场开闭、Session 交接、点差、流动性、新闻窗口、报价跳变和可成交性会否使挂单/止损语义失真？ | 市场状态、Session、新闻/日历时效、报价/点差/失败标记（可得时）。 | 不可交易/高影响事件未处理：阻断或 P1；不可得时明确未验证。 |
| 外部与跨资产反证 | DXY、US10Y、新闻/日历、社交或风险事件是事实支持、反证还是未知？其时间语义是否与技术假设一致？ | 来源、发布时间、as-of、事件倒计时、数据质量和 LLM 输入引用。 | 把旧事当未来、忽略已知高影响反证：P1；来源缺失：降级。 |
| LLM 决策质量 | LLM 是否以完整、同一 as-of 的事实生成候选？是否保留 Bull/Bear 分歧、反证、条件和不确定性，而非把高周期偏见变成入场指令？ | 每阶段 payload/输出/validator、事实 ID、冲突项、解析/回退来源、置信度含义。 | 编造、越权、静默回退或吞掉反证：P1；无法判断模型推理：证据不足。 |
| 授权、展示与人因 | Manager 是否仅授权已满足条件的计划？报告/UI 是否把方向偏见、条件计划、执行就绪、仓位 scale 和可靠度清楚分开，避免读者误作市价指令？ | ManagerDecision、signals、final decision、Narrative、UI/导出快照和不变量结果。 | 未触发候选被包装为执行、或高置信度掩盖条件：P1。 |
| 事后验证与校准 | 同类 setup 在冻结的 point-in-time 样本中是否记录过触发率、最大不利变动、止损/目标先后、成本和不同市场状态表现？ | 回测/前瞻 ledger、样本量、成本假设、市场状态标签和样本内/外边界。 | 无历史样本不等于策略错误，但不得把单次“看起来合理”称为已校准。 |

情景矩阵不是要求每天臆测未来，也不要求为所有维度虚构数据。没有数据时必须写“未验证”，并按该缺口降低计划的可执行状态和可靠度；只有数据明确表明不成立时，才报告为缺陷。

| 检查面 | 每日必须回答的问题 | 归档证据与离线复现 | 判定与处置 |
| --- | --- | --- | --- |
| 时间定位 | 用户、图表与归档所说的关键 K 线是哪一根？其交易所/UTC/本地时区、开收时间、是否已收盘为何？ | 归档 OHLC 的原始索引、显示时区、as-of 和转换后的时间；报告同时保留原始时区与转换结果。 | 把另一根 K 线当作被审对象、或将未收盘 bar 当确认结构，影响交易判断时为 P1；时区无法确认时为证据不足。 |
| 冲量与后续路径 | 最近一个与计划相关的冲量 K 线，其实体/全幅相对 ATR、成交量、收盘位置、突破的结构/价值区为何？之后是延续、回撤、首次回测、跌破/突破还是重新夺回？ | 对关键 bar 记录 OHLCV、`body / ATR`、`range / ATR`、成交量相对基线、突破前后至少若干根已收盘 K 线；用固定 OHLC 夹具复现事件序列。 | 报告声称“拒绝/突破/回测”却没有对应已收盘证据为 P1；指标可算但没有足够路径证据则降为候选/观察。 |
| 支撑/阻力角色转换 | 入场区在冲量后是已确认的支撑、阻力，还是仅为可能转换？若由支撑转阻力（或相反），何时以何种收盘规则跌破/夺回，是否已有首次回抽和拒绝？ | 将 FVG/OB/PA/SR 的来源、区间、方向与每根相关 K 线的 close/high/low 对齐；明确 wick 还是 close 是确认条件。 | 只因 LLM 或单一高周期标签就把相反的低周期已验证区域改称为阻力/支撑，为 P1；结构冲突未裁决时不得提升为执行建议。 |
| 止损与失效 | SL 是否真正使**该执行周期的交易假设**失效，而非只在入场区外？它是否越过相关 swing、价值区边界或确认 K 高/低，并有经验证的波动缓冲？ | 记录入场区两端/中位到 SL 的距离、5m/15m ATR 倍数、相关结构高低/VA 边界、spread/tick（可得时）和失效定义（wick/close）。 | 仅通过几何/R:R、但 SL 位于相关结构内或低于/高于未失效的关键边界时为 P1；缺少 ATR、结构或成本证据时为 P2，并降级为观察。 |
| 触发与执行就绪 | 当前是候选、观察、已触发还是已失效？计划要求的 5m/15m 拒绝、reclaim、CHoCH 或收盘条件是否真的发生在 as-of 前？ | `status`、`trigger_confirmed`、trigger evidence、最后 K 线、ManagerDecision、报告/UI 文案和 trace。 | `trigger_confirmed=false` 的候选被标为“执行/缩仓执行/可下单”，或执行文案掩盖条件，为 P1；仅有方向偏见而无触发证据时只能为条件计划。 |
| 多周期冲突与置信度 | 4h/1h 偏见、15m 结构、5m 触发、冲量方向、外部事件和反向区域是否冲突？报告是否把“方向偏见置信度”误写成“此刻可执行置信度”？ | Debate、technical facts、candles、Levels/Trader/Risk/Manager 输出和最终 Narrative 的逐字段对照。 | 未说明的实质冲突、或在未触发/冲量反向未裁决时给出高执行置信度，为 P1/P2；不能以模型自评替代证据。 |

每日审计必须输出一条可复核的**日内结构时间线**：`关键冲量 bar → 首次反应/回测 → 关键区被守住或跌破/夺回 → 当前价相对入场区 → 触发状态 → SL 相对结构与 ATR`。这条时间线只使用归档 as-of 前已收盘数据；它不等于未来表现预测。

除时间线外，每日 Ledger 必须附上该计划的**情景矩阵摘要**：市场状态、结构冲突、触发、失效、执行摩擦、外部反证、LLM/授权、展示语义与校准证据分别为何状态，以及使其无法成为执行就绪计划的首个阻断原因。这样冲量、假突破、区间反转、新闻跳变、流动性枯竭、模型幻觉和 UI 误导都走同一套可审计路径。

对短周期、窄止损策略，审核不得擅自规定一个未经回测校准的固定 ATR 阈值。相反，必须说明实际 ATR 倍数、结构边界和成本可见性；若无法证明该缓冲足以匹配所声称的 5m/15m 交易假设，结论应是“证据不足/观察”，而不是“几何合格即可执行”。

#### 4.2.3 主张级证据裁决与技术依据资格

日内情景矩阵必须落实到报告的**每一条主张**，不能只在计划层写“结构冲突”后仍允许某个未裁决事实主导结论。凡被 Levels、Trader、Risk、Manager、Narrative 或 UI 用作方向、入场、止损、目标、仓位或置信度理由的 FVG、OB、PA/SR、swing、流动性、事件或跨资产事实，均须建立可复核的“主张—证据裁决记录”。

| 裁决步骤 | 审核必须记录 | 合格标准 | 不合格处置 |
| --- | --- | --- | --- |
| 主张提取 | 原文/结构化字段、引用阶段、影响的 `signal_id` 与它被用来支持的结论。 | 每个核心理由都能定位到报告或阶段输出，不能把 LLM 概括语当作无来源事实。 | 核心理由无法定位：P1；非核心解释缺失：P2/P3。 |
| 独立复算 | `fact_id`、来源、时间框架、as-of、原始 OHLC/外部输入、计算版本、区间/方向/索引。 | 审核可不依赖 LLM 文本而复算数值和时间语义。 | 计算、时区、索引或可见性错误：按交易影响定 P1/P2。 |
| 质量判定 | 形成 K 线、缺口/区域宽度相对 ATR、动量与阈值距离、触及/回补/失效状态、成本和执行周期的相关性。 | 质量字段完整；任何“强/关键/共振”措辞均有可定位指标或经校准策略规则支撑。 | 数值成立但质量未校准或接近门槛：不得升级为核心依据，标记“观察/证据不足”；若仍驱动执行结论：P2。 |
| 上下文裁决 | 同级和高/低周期反向 FVG/OB/PA、趋势、关键 K 线路径、事件与跨资产反证；冲突的方向、范围、优先级和处理结果。 | 反证被显式列出并说明为何不足以推翻主张，或主张被降级。 | 已知实质反证被遗漏、改名或由 LLM 静默覆盖：P1/P2。 |
| 引用资格 | `核心执行依据`、`辅助背景`、`仅供观察`、`不可引用` 四选一；以及允许它进入下游的条件。 | 只有“核心执行依据”可影响执行授权；资格必须与 Manager、Narrative/UI 的表述一致。 | 将“观察/不可引用”事实表述为共振、确认或可执行理由：P1。 |

其中“计算正确”与“有资格主导交易结论”是两个独立结论。以 FVG 为例：三根 K 线满足公式只证明该区间可被标记为 FVG；若区间很窄、动量仅边界通过、与反向高/低周期结构重叠、未回踩拒绝或未满足触发条件，它只能是辅助背景或观察，不能单独构成入场、窄止损或高执行置信度的依据。不得用固定且未经点时回测校准的 ATR 数值替代该判断；应记录实际数值、策略版本和校准边界。

每个主展示计划必须包含一份最小**主张—证据裁决表**：`claim_id → 原始事实/输入 → 独立复算 → 质量与反证 → 引用资格 → 下游阶段/Manager 授权 → 报告展示`。表中任何核心主张缺少“质量与反证”或“引用资格”，该计划不得标记为执行就绪；审核也不得写“无新增 P1/P2”。

**关系与蕴含验证：**除单个事实存在外，审核必须验证 LLM 所声称的**事实之间的关系**。凡使用“共振、重叠、紧邻、供给/需求区、支撑转阻力、确认、覆盖、突破/收回”等关系词的核心理由，必须以结构化字段记录：参与的 `fact_id`、各自区间/点位、方向、时间框架、as-of、关系类型、容差与独立计算结果。不得用 `rationale`、`anchor_level` 或其他自由文本代替关系证据；自由文本不得单独获得“核心执行依据”资格。

| 关系检查 | 必须独立验证 | 不合格处置 |
| --- | --- | --- |
| 区间关系 | FVG、OB、PA/SR、VA/POC 与入场/SL/TP 的重叠、间距、包含或不相交；不得把相距的区域写成共振。 | 关系计算与文本不符：P1；无法得到结构化关系：降级为辅助/观察。 |
| 方向与角色 | 每个区间的原始方向、当时的支撑/阻力角色、触及/回补/失效状态，以及角色转换所需的收盘或路径条件。 | 把 neutral/相反方向事实改称为确认性供给或支撑，且影响计划：P1。 |
| 时效与质量 | 事实形成时间、触及次数、是否被后续价格消耗、宽度/ATR、成交量与执行周期相关性。 | 陈旧或已失效事实未降级：P1/P2。 |
| 主张蕴含 | 输出中的“因此/共振/确认/可执行”是否能由已记录事实与关系推导；同一事实不能被在下游改写、扩大或重新命名。 | 无法由输入推出却进入 Levels、Trader、Risk、Manager 或 Narrative：P1。 |

每日审核应沿传播链逐项比对：`技术 Analyst 原始输入/输出 → Levels → Trader → Risk → Manager → Narrative/UI`。任一阶段新增、重述或强化技术理由时，都要重新验证其 `fact_id` 与关系；不能因上游有一个已解析的 reaction ID 就假定其中的全部文本已经被验证。

**降级项的 Issue 升级规则：**“降级为条件计划/观察”不是 Issue 的豁免。若降级项已经实际影响某个主展示/获批信号的选择、Risk/Manager 授权、仓位、执行就绪状态或 Narrative/UI 展示，并且能给出代码位置与最小复现（或明确的数据契约冲突），必须在 Decision 阶段按影响创建/更新 P1 或 P2：已知实质反证被遗漏、观察级事实被升格为执行理由、或因此误导执行语义为 P1；缺少质量校准、反证裁决、成本/波动证据或引用资格但仍影响策略为 P2。只有尚未影响任何实际决策、无法定位根因的覆盖缺口、环境阻断，或确实只能证明“证据不足”而没有行为错误时，才只写入 Runtime Ledger（P3/未验证边界）。

### 4.3 Agent 与 LLM 决策链

- **审计目标**：逐阶段验证 Analyst、Bull/Bear、Debate、Levels、Trader、Risk、Manager 与 Narrative 的 payload、schema、来源、错误、重试、延迟、回退与置信度。
- **关键不变量**：每个阶段都能识别实际来源（LLM、规则、缓存或回放）、解析状态和回退原因；结构化输出在进入下游前已验证；置信度不因回退、空输出或不完整证据而被夸大。
- **证据来源**：归档的阶段输入摘要/结构化输出、`stage_meta`、agent trace、模型与延迟、解析/验证错误、回退标记、配置快照。
- **离线测试与最小复现**：对无效 JSON、字段缺失、枚举越界、超时、重试耗尽、空响应、规则回退、不同阶段并发隔离和不可信置信度构造夹具。
- **失败分级**：未验证模型文本直接进入交易授权或伪造来源为 P0/P1；静默解析/回退、延迟/错误不可观测为 P2；模型能力或成本比较证据不足为 P3。
- **未验证边界**：每日归档审核不会新增模型调用，不能证明供应商当前可用性或最新模型质量；该部分仅由每周/发布前受控抽样覆盖。

#### 4.3.1 大模型输入输出专项审核

此专项审核的单位是一次**阶段调用/回放工件**，而不是仅检查最终自然语言。原始提示词、原始响应和敏感字段不得直接写入 Ledger 或 Issue；需要保留时仅保存脱敏副本、摘要、哈希和可访问的归档引用。

| 检查面 | 输入侧审核 | 输出侧审核 | 证据与处置 |
| --- | --- | --- | --- |
| 输入契约 | 记录 stage 名称、schema/prompt/template 版本、模型与参数、payload 字段清单/哈希、as-of、数据窗口、事实 ID、市场状态和上游来源；检查必填字段、类型、单位、时区与长度限制。 | 验证输出能回链到允许的输入事实；不得把提示中的示例、系统指令或未注册文本变成市场事实。 | 保存脱敏 payload 摘要、版本与输入哈希；字段错配、未来数据或跨运行 payload 串线按影响定为 P0/P1。 |
| 指令与上下文隔离 | 不可信新闻、社交文本和外部字段必须作为数据而非可执行指令；检查 prompt 拼接边界、引用标识和敏感信息最小化。 | 检查输出没有复述凭据、系统提示或把外部文本的指令当作授权。 | 注入“忽略规则”“虚构价位”等对抗夹具；绕过授权或泄露敏感信息为 P0。 |
| 结构化解析 | 检查请求/响应大小、超时、重试、停止原因、JSON/schema 版本及解析前后的字段差异。 | 验证 JSON 可解析、枚举/数值范围/单位/必填项正确；禁止解析器静默丢弃关键字段、自动补造值或接受截断输出。 | 保存原始响应哈希、解析结果、错误码和 repair/fallback 来源；无效输出进入下游为 P1，静默字段丢失为 P2。 |
| 语义与事实约束 | 检查输入事实是否完整、陈旧、相互矛盾或已降级，并将此状态传给模型。 | 逐项检查事实 ID、时间语义、方向、价格、SL/TP、条件、置信度与 Manager 授权；模型“合理解释”不能替代事实或交易几何验证。 | 用伪造/陈旧事实、反向价格、空证据和矛盾上下文夹具回放；越权计划或编造事实为 P1。 |
| 回退与来源 | 触发超时、传输错误、解析错误、schema 失败、预算耗尽与显式禁用，确认每一种路径都有来源标签。 | 验证规则、缓存、回放、空结果和降级文本不会伪装成实时 LLM 成功；置信度、可靠度和下游权限应随之收紧。 | 在相同 fixture 上逐一触发失败路径；错误来源、静默回退或错误高置信度为 P1/P2。 |
| 漂移与稳定性 | 固定输入、模型版本、参数和温度后比较重复调用；对单字段扰动、顺序扰动和缺失事实检查敏感性。 | 记录 schema 通过率、解析/回退率、延迟、token、事实覆盖、授权一致率和置信度分布；将模型/提示/数据变化与输出漂移分开归因。 | 仅每周或发布前受控真实调用；每日只比较归档工件。无共同条件时记录“不可比”，不判定模型回归。 |

每个实际调用或复用阶段的最小 I/O 审计记录应包含：`run_id`、stage、输入哈希与 as-of、上游事实 ID、模型/配置版本、响应哈希、解析/验证结果、fallback 来源、延迟/重试、置信度、下游消费者和最终授权状态。缺失原始内容时可保留上述元数据与脱敏摘要；但必须明确该缺失造成的验证边界。

### 4.4 交易计划与授权

- **审计目标**：验证候选信号到 Manager 决策、最终报告和 UI 的方向、`signal_id`、价格、SL/TP、R:R、仓位、失效条件与观察模式一致性。
- **关键不变量**：仅 Manager 明确批准的计划可被执行性叙事、报告和 UI 引用；方向与目标顺序符合交易几何；SL、当前价距离、R:R、仓位缩放与失效条件在所有消费者中一致；观察/拒绝状态不得伪装为可执行交易。必须区分 **计划获准**（可保留为条件方案）与 **执行就绪**（触发已确认、数据可执行、未失效且通过日内执行质量回放）；`trigger_confirmed=false` 不得被 “execution_authorized”“执行”“缩仓执行” 等无条件措辞掩盖。
- **证据来源**：`validated_plans`、signals、ManagerDecision、risk scale、报告计划、UI 决策摘要、授权/不变量审计结果。
- **离线测试与最小复现**：构造价格反向、SL/TP 几何无效、重复/错配 `signal_id`、目标乱序、Manager 拒绝但 Narrative 引用、scale 不一致、未触发候选被授权、SL 位于执行周期结构内及空计划场景。
- **失败分级**：未经授权或未触发信号、错误价格或方向被呈现为可执行为 P0/P1；风险缩放、失效条件、执行周期不匹配或观察模式丢失为 P1/P2；非执行性文案改进为 P3。
- **未验证边界**：审核不评估实际成交、滑点或策略盈利；MT5 连接/下单始终在审核范围外。

### 4.5 报告可信度与叙事

- **审计目标**：验证事实注册、证据溯源、不变量门禁、叙事价格/方向/授权校验、可靠度语义和降级展示。
- **关键不变量**：叙事只能引用已注册且仍有效的事实与 Manager 明确授权的计划；任何新增/替换/修改执行价位、方向或 `signal_id` 均须被门禁拒绝；可靠度应反映证据完整性而非文案流畅度；降级原因对用户可见。
- **证据来源**：fact registry、evidence provenance、invariant audit、Narrative、ManagerDecision、可靠度标签、黄金报告夹具与 UI 展示。
- **离线测试与最小复现**：注入伪造事实、陈旧事实、未授权计划、错误价位/方向、空顶层叙事、无效几何和相互矛盾的可靠度字段，验证拒绝与降级。
- **失败分级**：报告把未授权或编造交易计划呈现为批准建议为 P1；门禁/可靠度静默失效或错误降级为 P2；文案完整性缺口为 P3。
- **未验证边界**：通过叙事校验仅证明“未超出授权与事实”；不保证市场预测会实现。

#### 4.5.1 分析报告输出内容正确性专项

本专项将“输出正确”拆为可在生成时验证的内容正确性，和只能在未来市场结果中验证的预测有效性。前者是每日归档审核与发布门禁的硬要求；后者必须采用严格 point-in-time 的事后评估，不能被单次报告、流畅文案或模型自评替代。

| 正确性层次 | 必须检查的内容 | 证据与离线验证 | 审核结论 |
| --- | --- | --- | --- |
| 事实正确性 | 每一项可验证的市场、宏观、新闻、日历、结构和风险陈述均有事实 ID/来源/as-of；禁止把假设、模型意见或过期信息写成事实。 | 将报告句子/字段映射至 fact registry 与 provenance；对伪造、陈旧、来源冲突、未来事件和缺失事实夹具进行回放。 | 无事实支撑或时间错误的关键内容为 P1/P2；不能验证时标记“证据不足”，不能标为正确。 |
| 数值与交易几何正确性 | 现价、入场、SL、TP、R:R、仓位/scale、`signal_id`、方向、目标顺序、失效条件和单位/精度与已验证计划一致；SL 还须与执行周期结构、ATR 缓冲和成本可见性相容。 | 从 `validated_plans`、ManagerDecision、报价快照和日内结构时间线逐字段比对；注入反向方向、错价、tick 舍入、目标乱序、结构内止损和无效几何。 | 错误数值、结构内失效位或方向被呈现为执行建议为 P1；非执行展示的格式/精度误差为 P2。 |
| 授权与语义正确性 | 结论、行动建议和置信度不能超出 Manager action、批准信号与风险 scale；必须清楚区分方向偏见、条件计划与执行就绪；观察、拒绝、降级和不确定性必须保留原义。 | 使用 Manager 拒绝、空授权、未触发候选、低可靠度和规则回退的黄金报告；验证 Narrative/UI 不会转换成隐式买卖建议。 | 新增/替换授权计划、将候选/观察状态写成交易建议，或把方向置信度写成执行置信度为 P1。 |
| 跨展示面一致性 | 顶层摘要、章节、表格、图表标注、agent trace、导出报告和 UI 卡片引用同一决策版本、同一价位与同一可靠度。 | 针对同一 run_id 生成各展示面快照，逐字段比较；构造章节滞后、图表错价、重复信号和截断文本夹具。 | 可能改变用户交易理解的不一致为 P1/P2；不影响语义的排版问题为 P3。 |
| 完整性与可理解性 | 必填风险提示、数据 as-of、来源/模式、失效条件、降级原因和未验证边界在适当位置可见；空字段、截断、未知值不得被默认值掩盖。 | 使用缺数据、解析失败、长文本截断和空计划夹具；检查显示状态与底层 meta 一致。 | 隐藏关键降级或遗漏风险条件为 P1/P2；非关键文案缺口为 P3。 |
| 预测有效性（事后） | 方向、触发、失效、目标与风险语义在预先冻结的规则下，与后续点位/结果比较；同时统计样本数、命中、最大不利变动、延迟和成本。 | 仅用不含未来数据的回测/前瞻记录；分离样本内、样本外、市场状态和重叠信号，保留未触发/未结案样本。 | 输出为“结果待验证”或“受控样本表现”，不得将盈利结果反推为当时内容正确；泄漏未来数据为 P0/P1。 |

每个最终报告应在审计产物中获得一个明确状态：**已验证正确**（事实、数值、授权和展示面均通过）、**已拒绝/已降级**（门禁发现错误且不作为建议展示）、**证据不足**（不能验证，附原因）或 **结果待验证**（仅适用于预测有效性）。这些状态必须与报告的可靠度语义一致，不能由模型自行声明。

### 4.6 回放、归档与回测

- **审计目标**：验证归档完整性、schema/版本兼容性、最新可加载归档的深度审核、成功基线漂移比较、point-in-time 与回测数据泄漏防护。
- **关键不变量**：归档应包含可定位的代码/配置/输入摘要、关键阶段产物、ManagerDecision、最终报告与测试/审计摘要；回放不得混入未来数据或当前环境默认值；基线比较须声明共同字段与不可比条件。
- **证据来源**：archive manifest、schema 版本、HEAD、as-of、输入哈希/摘要、阶段工件、replay 日志、backtest 数据切片与运行 ledger。
- **离线测试与最小复现**：加载旧 schema、缺失阶段、损坏字段、未定位 HEAD、未来 K 线/事件注入、同输入重复回放和归档兼容夹具。
- **失败分级**：回测/回放使用未来数据或归档篡改不被发现为 P0/P1；关键产物缺失、版本兼容或漂移比较失真为 P2；历史归档覆盖不足为 P3。
- **未验证边界**：一个归档只证明该次输入和环境下的行为，不能代表实时市场、不同模型版本或不同数据源。

### 4.7 UI、运行与安全

- **审计目标**：验证用户能看见决策链、错误和降级状态；并发运行互相隔离；LLM 传输超时、凭据日志和 MT5 边界受控。
- **关键不变量**：UI 不得隐藏“规则/回放/LLM”来源、失败或观察状态；并发请求的 trace、信号和报告不得串线；凭据不得进入日志、归档或 Issue；审核流程不得发出交易指令。
- **证据来源**：UI 渲染状态、agent trace、请求/运行 ID、脱敏日志、异常路径、环境变量引用位置和 MT5 保护开关。
- **离线测试与最小复现**：模拟并发运行、LLM 超时/传输错误、脱敏检查、空/降级 trace、重复渲染及禁用交易环境。
- **失败分级**：凭据泄露、越过 MT5 边界或跨运行混用交易信号为 P0；错误/降级隐藏且影响决策为 P1/P2；可观测性改进为 P3。
- **未验证边界**：离线测试不能穷尽浏览器、代理、供应商网络及用户权限组合；应保留运行环境条件。

### 4.8 测试与治理

- **审计目标**：保证每项结论具有可复跑的离线测试或最小复现，并以根因去重同步治理记录。
- **关键不变量**：绿灯测试只能证明覆盖范围内的行为；失败必须归类为代码、数据、传输、解析、验证、回放兼容或环境；同一根因不重复建 Issue；P3 和证据不足项不伪装成缺陷。
- **证据来源**：测试命令与版本、失败输出、夹具、覆盖模块、最小复现、Runtime Ledger、GitHub Issue 与去重搜索结果。
- **离线测试与最小复现**：运行编译检查、受影响单元/场景测试、归档回归与跨模块不变量测试；发布前遵循[测试策略](../testing/strategy.md)的完整门禁。
- **失败分级**：阻断授权/安全/未来数据防护的测试失败按其业务影响定为 P0/P1；回归覆盖不足为 P2/P3。
- **未验证边界**：没有可控复现的异常不得提升为根因；应写明所需样本、环境或外部状态。

## 5. LLM 阶段检查表

每次涉及 LLM 的审核必须为每个实际经过的阶段填写“输入 → 输出 → 验证 → 回退 → 下游授权”。阶段不存在、被跳过或由回放复用同样需要记录原因；不得把规则输出标记为 LLM 输出。

| 阶段 | 输入检查 | 输出与验证 | 回退与可观测性 | 下游授权检查 |
| --- | --- | --- | --- | --- |
| Analyst（技术、基本面、新闻、情绪） | as-of、来源时间、市场状态、输入摘要和数据缺失标记。 | schema、事实引用、时间语义、置信度与来源一致。 | 记录模型、延迟、重试、解析错误、规则/缓存/空输出回退。 | 只能作为 Bull/Bear、Debate 和 Levels 的证据，不能直接授权交易。 |
| Bull / Bear | 共享 Analyst 事实及相同 as-of；禁止各自引入未注册事实。 | 方向论据、反证、条件和置信度可解析且可追溯。 | 对单边失败明确标记，不能静默用另一侧文本补齐。 | 仅向 Debate 提供立场，不能产生执行计划。 |
| Debate | Bull/Bear 的来源、条件和冲突点。 | 结论应保留争议、证据权重与不确定性，而非覆盖输入事实。 | 记录空输入、冲突、超时及保守回退。 | 可影响 Levels/Trader 的分析上下文，不能越过 Risk/Manager。 |
| Levels | 已验证的 OHLC/结构事实和 Debate 上下文。 | 入场、SL、TP、失效条件、方向与点位来源符合 schema/交易几何。 | 无效点位必须被验证器拒绝或降级，并保留原因。 | 仅形成候选 levels，须由 Trader、Risk、Manager 继续审核。 |
| Trader | Levels、市场状态、候选信号及约束。 | 信号选择、`signal_id`、方向、价格与理由结构化且可验证。 | 记录选择失败、重复 ID、无候选和规则回退。 | 仅提交候选计划给 Risk；不得直接成为报告批准计划。 |
| Risk | Trader 候选、风险约束、账户/市场边界（如可用）。 | scale、风险理由、拒绝/观察条件和置信度可解析。 | 超时或缺失时标明保守行为，禁止默认为完整风险批准。 | 仅向 Manager 提供风险意见；最终授权仍以 Manager 为准。 |
| Manager | 已验证计划、Risk 结果、授权约束与证据完整性。 | action、批准的 `signal_id`、scale、confidence、原因与观察/拒绝状态一致；对未触发候选只能授权为条件计划，不得声明执行就绪。 | 解析/验证失败必须关闭授权或明确降级，不能生成隐式批准。 | ManagerDecision 是报告、UI 与任何执行性表述的唯一授权源，但不替代 `trigger_confirmed` 和日内执行质量门禁。 |
| Narrative | ManagerDecision、fact registry、evidence provenance 与可靠度状态。 | 叙事中的方向、`signal_id`、价格、SL/TP、条件和可靠度通过不变量校验；说明关键冲量、角色转换、触发证据和止损失效逻辑。 | 任何越权、伪造、几何错误、结构内止损或条件被抹去都应被拒绝/降级并可见。 | 不得新增或替换计划；候选必须显式标为“等待触发”，不能以执行标题或高执行置信度替代条件。 |

### 5.1 LLM 对技术主张的引用审计

LLM 的“理由”不是证据本身。对于每个被 LLM 用来支持最终计划的技术主张，审核必须从输出反向追到第 4.2.3 的裁决记录，并检查它是否遗漏了关键反证、扩大了事实的强度，或把“存在”改写成“确认/共振/可执行”。

| LLM 行为 | 必须验证 | 审核判定 |
| --- | --- | --- |
| 引用技术事实 | 每个 FVG/OB/PA/SR/流动性/事件结论均有 `fact_id` 或等价可复算来源；价格、方向、时间框架和 as-of 一致。 | 不可追溯的关键引用为 P1；非关键引用为 P2/证据不足。 |
| 评价证据强度 | “强”“关键”“共振”“确认”“高置信度”等措辞有质量指标、反证处置和策略/校准版本支撑。 | 事实正确但被无依据升格为执行依据：P2；若由此改变 Manager/报告执行结论：P1。 |
| 处理冲突 | 同一位置的反向区域、低周期路径、未触发状态、事件和成本限制被保留，不能被单边叙事吞没。 | 已知实质反证被遗漏或反向命名：P1/P2。 |
| 下游授权 | Levels/Trader/Risk/Manager 只能使用具有明确引用资格的主张；Narrative/UI 不得重新强化被降级事实。 | 观察级事实进入执行授权，或叙事绕过资格标记：P1。 |

| 输入—输出蕴含与传播 | 对每个实际经过阶段保存或抽取原始/脱敏 prompt、结构化输入、原始输出、解析后对象与下游消费字段；逐项比较新出现、被改写或被强化的技术理由。 | 任何阶段把“存在的事实”扩写为未经验证的“共振/确认/供给/可执行”，或下游继续采用该扩写而影响授权：P1。 |

每日归档审核的 LLM I/O 最低动作不是只看 stage 成功、schema 或最终摘要：必须对每个主展示计划抽取其完整理由链，至少覆盖 `技术 Analyst → Levels → Trader → Risk → Manager → Narrative`。记录每一跳的输入事实、输出主张、解析/validator 结果、资格变化与授权影响；若归档只保留摘要而缺少实际输入/输出，必须明确该链条为“证据不足”，不能用模型 self-confidence 代替。

每日回放审核应至少抽取每个主展示计划的所有核心理由，而不是只抽查一个“看起来合理”的理由。若归档未保存足以完成反向追溯的引用或质量字段，结论应是“证据不足/不可执行”，并将归档可观测性缺口按影响记录为 P2/P3；不得以 LLM 自述补足。

## 6. Runtime Ledger 与 GitHub Issue 规则

### 6.1 每日 Runtime Ledger

每日审核无论是否发现缺陷均更新滚动 Ledger，并保留最近 14 次摘要。每条至少包含：

- 审核类型（每日归档、每周受控 LLM、发布前）、启动快照 HEAD/分支、目标归档 ID/时间与数据 as-of；
- 归档可加载性、基线 ID（如有）、复用/实际复跑阶段、是否有真实 LLM 调用、模型/token/耗时（仅实际调用时）；
- 最终方向、获批 `signal_id`、Manager action/scale/confidence、报告/授权链状态；
- 日内结构时间线（关键 K 线的原始时区与本地时区、冲量/首次回测/跌破或夺回、入场区角色、触发状态）；入场区两端/中位到 SL 的距离、5m/15m ATR 倍数、相关结构边界与冲突结论；
- 每个主计划的日内决策情景矩阵状态、首个阻断/降级原因、已知反证与未验证数据；不得只写“几何通过”或“LLM 高置信度”；
- 每个主展示计划的核心技术主张裁决：至少列出 `claim_id`、独立复算、质量/阈值边界、反证、引用资格及其是否实际影响 Manager/叙事；数值正确但不具备核心依据资格时，必须单列而非并入“结构通过”；
- 每个主展示计划的 LLM 理由传播表：`技术 Analyst 输入事实/输出主张 → Levels → Trader → Risk → Manager → Narrative/UI`，并列出每次新增或强化的关系主张、其 `fact_id`/区间关系验证结果、首次失真阶段和最终授权影响；不得仅以“各阶段成功”代替。
- 通过/失败的测试与最小复现、P0–P3 汇总、环境阻断、未验证边界；
- 对比基线时的 Analyst bias、debate、levels、Trader、Risk、Manager 漂移，以及不可比条件。

“无新增 P1/P2”同样是有效结论，但必须保留检查范围与证据，不能仅写结果。

### 6.2 Issue 创建与去重

只有 P0/P1/P2 的**证据完整根因**才创建或更新 Issue。写入前按“根因 + 文件/符号”搜索现有开放与历史 Issue；已有同根因时追加新证据、复现范围和回归结果，而不是重复创建。

每个可交给修复者的 Issue 应包含：

1. 严重度、触发条件和用户/交易影响；
2. 代码位置（文件、符号、行号或稳定定位方式）；
3. 运行证据、归档/输入条件和最小复现；
4. 预期与实际行为，以及其属于数据、传输、解析、验证、兼容、环境还是代码；
5. 最小整改方向与明确的验收测试；
6. `daily-audit` 标签（标签权限失败不应中止审核）。

P3、环境阻断、供应商不可用和证据不足项仅写入 Ledger，并标注下一步需要的样本或验证条件。Issue 不是一次审核的替代报告，也不能代替根因去重。

## 7. 审核完成与维护验收

一次审核可以宣告完成，当且仅当：

- 已声明审核节奏、输入、启动快照、归档/基线与可比性；
- 所有覆盖模块均有目标、不变量、证据、离线测试/最小复现、分级和未验证边界记录；
- 所有实际经过的 LLM 阶段均完成本蓝图的五段检查表，或明确未经过/复用原因；
- 发现项已按 Ground → Evidence → Decision 分类，P0/P1/P2 已去重同步，P3 与环境边界未被误报；
- Ledger 已更新，且发布前审核满足[测试策略](../testing/strategy.md)和相关架构门禁。

当流水线 schema、授权边界、事实注册、不变量、归档格式或测试分层变化时，应在同一变更中复核本蓝图的链接和检查项。具体实现事实仍应更新到相应架构文档；本蓝图只维护审核契约与验收口径。
